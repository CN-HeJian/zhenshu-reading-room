# 枕书：微信读书自动同步设计

## 1. 背景与目标

“枕书｜我的阅读札记”当前第 3 版使用静态演示数据。项目运行在 Sites 的 vinext/Cloudflare Worker 运行时，生产环境已经保存 `WEREAD_API_KEY` secret，站点访问策略为仅所有者可访问，但尚未配置 D1 数据库。

本次改造的目标是让网站在不暴露微信读书密钥的前提下，自动同步并展示以下个人数据：

- 书架，包括电子书、专辑/有声书和文章收藏入口；
- 电子书阅读进度与累计阅读时长；
- 个人划线、想法/点评，以及只能统计但不能导出内容的书签数量；
- 阅读统计，包括阅读时长、阅读天数、主要阅读书目和可用的偏好数据。

同步任务每天北京时间 23:30 运行。站点保留“立即同步”入口，并在错过定时任务时于下一次访问时补同步。所有已同步历史数据长期保存，不生成日报、周报或 AI 总结。

## 2. 已确认的约束

- 保留现有视觉语言和页面结构，不重新设计网站。
- `WEREAD_API_KEY` 只在服务端读取，不下发到浏览器、不写入数据库、不进入日志。
- 生产站点维持仅所有者可访问。同步入口不得在公开或共享状态下无附加保护地运行。
- 微信读书网关统一使用 `POST https://i.weread.qq.com/api/agent/gateway`。
- 每次网关请求都上报 `skill_version: "1.0.3"`，业务参数与 `api_name` 平铺在请求体顶层。
- 任一响应出现 `upgrade_info` 时，立即停止当前同步阶段并显示升级提示，不用旧字段继续写入。
- 所有 Unix 时间戳按北京时间展示为日期或日期时间；所有阅读时长源字段按秒处理。
- 本次只读取微信读书数据，不向微信读书写回内容。

## 3. 方案选择

采用“Sites 服务端同步 + D1 持久化 + Codex 定时任务”的组合：

1. 网站服务端实现唯一的同步服务和受保护的同步接口。
2. D1 保存规范化的当前数据、变化历史和同步运行记录。
3. Codex 自动化任务每天北京时间 23:30 获取该 Sites 项目的短期访问凭据，触发同步并跟进至终态。
4. 浏览器的“立即同步”和访问时补同步调用同一服务，不维护第二套同步逻辑。

选择该方案的原因是当前 Sites 项目没有暴露 Cron Trigger 配置。Cloudflare Worker 的 `scheduled()` 处理器只有在同时配置 Cron Trigger 后才会运行，因此不能仅靠修改当前 Worker 代码保证 23:30 后台执行。

## 4. 系统边界与组件

### 4.1 微信读书网关适配器

`lib/weread/` 负责请求构造、鉴权、响应校验、分页和字段映射。上层同步服务只接触稳定的内部模型，不依赖微信读书原始字段名。

首版使用以下读取能力：

- `/shelf/sync`：书架电子书、专辑/有声书和文章收藏入口；
- `/book/getprogress`：电子书当前章节、进度百分比、累计阅读时长和读完时间；
- `/user/notebooks`：按 `count + lastSort` 游标遍历全部有笔记的书；
- `/book/bookmarklist`：单本书的划线内容；
- `/review/list/mine`：按 `synckey` 遍历单本书的个人想法与点评；
- `/readdata/detail`：本周、本月、本年和总计统计。

网关适配器必须落实这些口径：

- 书架可见条目总数为 `books.length + albums.length + (mp 非空 ? 1 : 0)`；
- 阅读进度是 0–100 的整数，`1` 表示 1%，只有 `100` 表示读完；
- 笔记统计数为 `reviewCount + noteCount + bookmarkCount`；
- 可展示的笔记内容为划线内容加想法/点评内容，书签只展示数量；
- `totalReadTime`、`readTimes`、`dailyReadTimes`、`dayAverageReadTime` 等时长字段单位均为秒；
- `/user/notebooks` 翻页只使用顶层 `lastSort`，`/review/list/mine` 翻页只使用其返回的 `synckey`。

### 4.2 同步编排器

`lib/sync/` 负责创建同步任务、获取互斥锁、分阶段执行、提交 D1 写入和汇总结果。同步来源分为 `scheduled`、`manual` 和 `catch_up`，三者共用同一编排器。

一次完整同步按以下顺序执行：

1. 同步书架并标记本次仍存在的条目；
2. 同步全部电子书的当前阅读进度；
3. 遍历全部笔记本，更新笔记统计，再同步每本书的划线和个人想法/点评；
4. 同步本周、本月、本年和总计阅读统计；
5. 计算汇总指标，完成同步运行记录并释放锁。

外部请求数量随书架和笔记本数量增长。编排器采用限并发和可续跑设计：每次处理固定大小的书籍批次，保存阶段与游标；自动化任务或浏览器在收到 `running` 状态后继续调用续跑接口，直至 `success`、`partial_success` 或 `failed`。这样避免单次 Worker 请求超时，也不会因为重试而重复数据。

### 4.3 服务端接口

- `POST /api/sync`：创建新同步，或在已有同步运行时返回当前运行 ID；请求体只允许 `manual`、`scheduled`、`catch_up` 三种来源。
- `POST /api/sync/:runId/continue`：执行下一批同步工作，终态任务不可再次推进。
- `GET /api/sync/:runId`：返回进度、阶段、各类新增/更新数量和安全的错误摘要。
- `GET /api/sync/status`：返回上次成功时间、当前运行状态和是否需要补同步。

手动和访问时补同步依赖 Sites 的所有者身份。定时任务每次从 Sites 获取短期访问凭据后调用相同接口，不在任务定义中保存长期站点访问令牌。如果站点访问策略不再是仅所有者，自动任务必须停止并要求配置独立的服务间 secret，不能默认信任公共请求。

### 4.4 Codex 自动化任务

自动化任务按 `Asia/Shanghai` 每天 23:30 运行，固定绑定站点项目 ID `appgprj_6a53439c8b0c8191b9d764a45038c664`。每次运行：

1. 检查站点仍为仅所有者可访问；
2. 获取短期 Sites 访问凭据；
3. 触发 `scheduled` 同步；
4. 按服务端返回状态调用续跑接口，直到终态或达到安全执行上限；
5. 仅在失败、部分成功或需要 skill 升级时向用户报告，成功运行不生成阅读报告。

如果本地 Codex 自动化未运行，网站在下一次访问时发现“上次成功同步时间早于最近一个北京时间 23:30”，提示并自动发起 `catch_up`。用户也可随时手动发起同步。

## 5. 数据模型

D1 使用 Drizzle 管理迁移。所有源 ID 以字符串保存，避免数值精度或未来格式变化。

### 5.1 `shelf_items`

保存电子书、专辑/有声书和文章收藏入口的当前状态。

核心字段：内部 ID、`source_type`、`source_id`、标题、作者/演播者、封面、分类、是否读完、是否置顶、是否私密、源更新时间、最近阅读/收听时间、首次见到时间、最后见到时间、离架时间。`source_type + source_id` 唯一。

本次未出现在书架响应中的旧条目只设置离架时间，不物理删除。

### 5.2 `reading_progress_current`

每本电子书一条当前进度：`book_id`、章节 UID、章节内偏移、进度百分比、累计阅读秒数、是否开始阅读、源更新时间、读完时间和最近同步时间。

### 5.3 `reading_progress_history`

只在进度、章节、累计时长或读完状态发生变化时写入快照。字段包含书籍 ID、上述可见进度字段、源更新时间和采集时间。所有快照长期保留。

### 5.4 `notebook_summaries`

每本书保存 `review_count`、`highlight_count`、`bookmark_count`、阅读进度、标记状态和最近笔记排序值。页面总笔记数通过三类计数求和，不把 `noteCount` 误当总数。

### 5.5 `highlights`

以 `bookmarkId` 唯一保存个人划线：书籍 ID、章节 UID、章节标题、原文、范围、颜色、创建时间、最后见到时间和移除时间。具备章节与范围时生成微信读书深度链接。只有单本书划线列表完整同步成功后，才把本次未出现的旧划线软标记为已移除，不物理删除历史内容。

### 5.6 `personal_reviews`

以 `reviewId` 唯一保存个人想法与点评：书籍 ID、内容、章节名、评分、是否读完、创建时间、最后见到时间和移除时间。只有源响应同时提供章节 UID 与范围时才生成定位链接；整本书评不伪造链接。只有单本书点评列表完整同步成功后，才把本次未出现的旧点评软标记为已移除。

### 5.7 `reading_stat_snapshots` 与 `reading_time_buckets`

`reading_stat_snapshots` 按同步时间和统计模式保存本周、本月、本年与总计的汇总：基准时间、总阅读秒数、有效阅读天数、自然日均秒数、与上期对比，以及可用的排行/偏好结构。

`reading_time_buckets` 将接口返回的分桶明细规范化为日期或月份，用于近 7 天、近 30 天和累计趋势展示。相同周期与分桶开始时间幂等更新，旧快照不删除。

### 5.8 `sync_runs` 与 `sync_lock`

`sync_runs` 保存运行 ID、来源、状态、当前阶段、分页/批处理游标、开始/结束时间、各类新增/更新数量和安全错误码。状态只允许 `running`、`success`、`partial_success`、`failed`。

`sync_lock` 使用单行租约和到期时间防止并发同步。锁到期后可由新运行接管；所有写入仍以源 ID 幂等，保证崩溃重试安全。

## 6. 页面与交互

不改变现有视觉风格，只替换静态数据并增强现有控件：

- 顶部继续保留“我的书架 / 阅读批注”导航；
- “导入微信读书内容”改为“立即同步”；
- 同步弹窗显示当前阶段、进度、上次成功时间、新增/更新数量及失败提示；
- 首页藏书数使用包含电子书、专辑和文章收藏入口的书架总口径；
- 首页批注数使用想法/点评、划线与书签的统计总和；
- 首页阅读时长从阅读统计读取，并格式化为小时/分钟；
- 书架卡片展示真实封面、作者、状态、进度和最近阅读时间；专辑卡片明确标注“有声书”；
- 阅读批注按时间倒序合并划线与想法/点评，书签只显示数量；
- 现有搜索框在书架页搜索书名/作者，在批注页搜索书名、原文和想法；
- 现有“本周共读”区域改为“阅读足迹”，复用当前版式展示近 7 天、近 30 天和累计阅读数据；
- 有足够定位信息的书籍、章节和划线附上微信读书深度链接。

首次加载先显示 D1 中最近一次成功数据，不等待远程同步。若需要补同步，页面在加载后发起任务并显示非阻塞状态；同步成功后刷新受影响区域。

## 7. 错误处理与数据保护

- 未配置或失效的 `WEREAD_API_KEY`：同步失败，保留旧数据，界面显示鉴权提示，不返回密钥相关内容。
- `upgrade_info`：当前阶段立即停止，运行标记失败并展示升级说明，不继续解释或写入可能变更的字段。
- 限流、网络错误和 5xx：当前批次有限次指数退避；超过上限后标记部分成功或失败，保留已提交批次和续跑游标。
- 单一能力失败：其他独立能力可继续；最终状态为 `partial_success`，明确指出失败类别。
- 分页游标重复或不前进：停止该分页，记录结构化错误，防止无限循环。
- 数据缺字段：可选字段按缺失处理；稳定 ID、必要文本或时间字段缺失时跳过该记录并计入告警。
- 并发触发：只保留一个运行任务，其他请求返回该运行 ID。
- 旧数据保护：同步前不清空表；仅在一个书架阶段完整成功后才标记未出现条目为离架。
- 日志：只记录请求类别、计数、耗时和安全错误码，不记录 Authorization、完整原文或个人想法内容。

## 8. 测试与验收

### 8.1 自动测试

- 网关请求测试：URL、Authorization、`skill_version`、顶层参数和平铺分页游标正确；
- 书架口径测试：电子书、专辑和 `mp` 均计入总数，公开/私密计数正确；
- 进度测试：`1` 显示为 1%，`100` 才显示读完，时长按秒格式化；
- 笔记测试：完整遍历 `lastSort` 与 `synckey`，笔记总数公式正确，书签内容不会被伪造；
- 统计测试：总时长优先使用 `totalReadTime`，7/30 天展示使用规范化分桶，不混淆自然日均和阅读日均；
- 数据库测试：迁移可重复执行，幂等 upsert、变化快照、离架标记和长期历史均正确；
- 编排测试：互斥锁、租约过期、批次续跑、重复触发和部分失败状态正确；
- 权限测试：非所有者和缺少 Sites 访问凭据的同步请求被拒绝；
- 页面测试：无数据、同步中、成功、部分成功和失败状态都能渲染，搜索和深度链接正确。

### 8.2 上线验收

1. 生产构建、lint 和自动测试全部通过。
2. 用真实账号执行一次手动完整同步，确认密钥不出现在浏览器网络响应和日志中。
3. 对照微信读书核验书架总数、至少三本书的进度、至少两本书的划线/想法和本周/本月/累计时长。
4. 连续执行两次同步，第二次不产生重复记录；修改一个进度后只增加一条进度历史。
5. 模拟网关失败和 `upgrade_info`，旧数据仍可访问且提示清晰。
6. 检查桌面端和移动端现有样式，确保静态演示内容已全部替换。
7. 创建并试运行北京时间 23:30 的 Codex 自动化任务；验证错过运行后的访问补同步和“立即同步”均可完成。

## 9. 非目标

- 不生成日报、周报、AI 总结或阅读建议；
- 不同步公开热门划线、他人点评或划线下的公共评论；
- 不导出微信读书当前未提供的书签正文；
- 不同步 `mp` 入口内的具体文章内容；
- 不向微信读书写回笔记、书架或阅读进度；
- 不改为公共站点，不增加多用户账号系统；
- 不承诺实时同步，自动刷新粒度为每日一次并带访问补偿。

## 10. 完成标准

当生产站点能够以真实微信读书数据替换全部静态演示数据，每天 23:30 自动同步，访问时可补同步，用户可手动同步，所有历史长期保留，错误不会破坏旧数据，并通过第 8 节验收项时，本功能完成。
